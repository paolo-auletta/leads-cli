# Contact Discovery And Apollo Enrichment

Date: 2026-06-22

## Goal

The next step of the tool should start from already selected, verified, and enriched companies and
find the right people inside them.

This is not the same problem as company discovery.

The company phase answers:

> Which companies are in scope?

The contact phase answers:

> Which people at those companies match the requested roles, and how do we get usable contact
> details for them?

The user should be able to say things like:

- "for all of these companies find the manager"
- "for companies X and Y, find the project manager, information manager, and estimators"
- "for the selected Texas contractors, find owners and estimators"

The main inputs therefore become:

- a set of already-approved companies;
- one or more requested role targets.

## Core product decision

Keep the same two-step shape:

1. `contacts discover`
2. `contacts enrich`

But the responsibilities must be very clear.

### Contact discovery

Use the live web to identify who currently works at the company and whether they match the
requested role.

### Contact enrichment

Use Apollo only to fetch deliverability fields such as email and phone for already-discovered
people.

Apollo must not be trusted as the primary source of current employment or current title, because
the client explicitly says it can be stale by around 6 months.

That means Apollo is not the discovery engine.
It is the contact-channel enrichment engine.

## Official Apollo API findings

I checked Apollo's current official docs on 2026-06-22 to ground the implementation design in the
real API surface.

Most relevant endpoints:

- `POST /api/v1/people/match`
- `POST /api/v1/people/bulk_match`
- `GET /api/v1/webhook_result/{request_id}`
- `POST /api/v1/usage_stats/api_usage_stats`

Important product facts from the docs:

- `people/match` enriches 1 person.
- `people/bulk_match` enriches up to 10 people per call.
- Apollo matches better when you pass stronger identifiers such as `linkedin_url`, full name, and
  company domain.
- the normal people search endpoint is for prospecting, not for contact-channel enrichment, and it
  does not return emails or phone numbers.
- `reveal_phone_number=true` is asynchronous and requires `webhook_url`.
- waterfall enrichment for email and/or phone is also asynchronous and returns a `request_id`.
- Apollo exposes `GET /webhook_result/{request_id}` so we can poll results if webhook delivery is
  missed.
- Apollo's usage endpoint can report rate limits, but it requires a master API key.

Implementation consequence for this project:

- Apollo should stay in `contacts enrich`, never in `contacts discover`.
- contact enrichment should use our already-discovered identity as the input to Apollo, not the
  other way around.
- the best Apollo payload for this project is usually:
  - `linkedin_url`
  - `first_name`
  - `last_name`
  - `domain`
  - `organization_name`
- batching should use `bulk_match` in chunks of 10.
- phone enrichment cannot be treated as a simple sync field fetch; it needs webhook-aware run
  handling.
- for CLI ergonomics, the clean design is:
  1. submit Apollo enrich requests
  2. store `request_id`
  3. poll Apollo results
  4. merge final email / phone outputs into the run artifacts

## Why this split is correct

If we let Apollo drive contact discovery, we recreate the same problem the client already has:

- stale company assignment;
- stale role/title;
- false positives from old employment records.

If we let the live web drive discovery first, we can answer the more important question:

> Is this person currently the right person at this company?

Then Apollo becomes useful only after identity and current-company fit are already established.

This is the same design principle used in the company flow:

- discover with the best current source;
- enrich later with the best structured source for missing fields.

## Recommended product shape

```text
enriched company run
  ->
contact search spec
  ->
contacts discover
  ->
accepted / review / rejected people
  ->
contacts enrich
  ->
email / phone / Apollo trace
  ->
ready outreach list
```

## The right starting input

Contacts should not be searched against arbitrary companies typed manually every time.

The normal starting point should be a completed company enrichment run, because that gives us:

- a reviewed company list;
- root domain;
- normalized company name;
- target vertical;
- geography;
- independence context;
- a clean company artifact trail.

So contact discovery should usually accept one of these scopes:

1. all companies from an enrichment run
2. selected domains from an enrichment run
3. selected company names/domains from an enrichment run

This keeps contact search anchored to known-good companies.

## Spec should stay JSON-first

As with company discovery, the contact workflow should use a structured input file generated by the
agent skill from the user's natural-language request.

Recommended canonical format:

`contact_search_spec.json`

Why JSON again:

- stricter than YAML;
- easier to validate;
- easier for structured LLM output;
- easier to version with schema changes;
- easier to use deterministically in code.

YAML can be supported later if useful for humans, but JSON should remain the contract.

## Proposed contact spec

```json
{
  "version": 1,
  "company_source": {
    "enrichment_run_id": "company-enrich-a1b2c3d4e5f6",
    "bucket": "ready",
    "domains": []
  },
  "roles": [
    {
      "key": "project_manager",
      "labels": ["project manager", "senior project manager", "pm"],
      "max_per_company": 2
    },
    {
      "key": "estimator",
      "labels": ["estimator", "chief estimator", "senior estimator"],
      "max_per_company": 2
    }
  ],
  "company_limit": null,
  "contact_limit": null,
  "current_only": true,
  "require_role_match": true,
  "prefer_one_per_role": false
}
```

## Why roles need structure

The user will ask for things like:

- manager
- owner
- estimator
- project manager
- information manager

These are not reliable as plain strings.

The agent skill should convert them into structured role targets with:

- a stable key;
- one or more title labels or synonyms;
- per-company caps.

This lets the CLI behave predictably.

Example:

- user says "manager"
- skill may ask itself which kind of manager is implied by the company vertical and prior context
- final spec could resolve to `general_manager`, `operations_manager`, or `project_manager`
- if ambiguity remains high, the agent should make that explicit before creating the spec

The CLI itself should not guess ambiguous business intent from raw language.

## Contact discovery redesign

Contact discovery should be the hard intelligence layer, just like company discovery.

It must answer three questions:

1. who is the person?
2. do they currently work at this target company?
3. do they match one of the requested roles?

If discovery cannot answer those reliably, contact enrichment should not happen yet.

## Contact discovery pipeline

```text
load enriched companies
  ->
build per-company role plan
  ->
search live web for named people and role evidence
  ->
normalize person candidates
  ->
evaluate current-company fit + role fit
  ->
accepted / review / rejected
  ->
save reusable people memory
```

## Recommended sources for contact discovery

Use current web sources that can reveal who actually works there now.

Source priority:

1. official company website team/about/leadership/staff pages
2. official company website project pages or office pages
3. LinkedIn company employee results or company leadership references found on the live web
4. company press releases or announcements
5. conference speaker pages, association member pages, or credible business pages

The key idea is not "find any page mentioning a person."

The key idea is:

> Find current evidence tying a person, a company, and a role together.

## Search strategy

For each company and role target, the system should generate focused search queries.

Examples for a company called `Acme Builders` and role `project_manager`:

- `"Acme Builders" "project manager"`
- `site:acmebuilders.com "project manager"`
- `"Acme Builders" estimator`
- `"Acme Builders" linkedin "project manager"`
- `"Acme Builders" team`
- `"Acme Builders" leadership`

This is similar to company discovery:

- LLM generates focused search queries
- Exa executes them
- results are normalized
- LLM evaluates candidates against the structured role target

## Candidate normalization for people

The normalized contact candidate should contain:

- `full_name`
- `first_name`
- `last_name`
- `current_company_name`
- `current_company_domain`
- `observed_title`
- `matched_role_key`
- `linkedin_url` if available
- `source_urls`
- `evidence_snippets`
- `location_hint`
- `discovery_confidence`

This gives enrichment a stable person record to work from later.

## Contact discovery decision model

As with companies, do not invent fake mathematical scoring too early.

Use a small number of explicit verdicts:

- `accepted`
- `review`
- `rejected`

And base them on structured reasoning:

### Accept when

- the person is clearly tied to the target company now;
- the title clearly matches a requested role;
- identity is specific enough to avoid confusion.

### Review when

- the company tie looks plausible but not strong enough;
- the title is adjacent but not exact;
- multiple people with the same name are plausible;
- one strong source exists but no corroboration exists.

### Reject when

- evidence points to a different company;
- the person appears to be a former employee;
- the role is too far from the requested one;
- the identity is too ambiguous.

## Very important rule about current employment

Discovery should care more about current employment than about contact channels.

For example:

- if the internet says Jane Smith is currently Project Manager at Acme Builders
- and Apollo says Jane Smith is at OldCo

then the discovered role/company truth should remain:

- current company from live web: Acme Builders
- Apollo company mismatch: stale enrichment conflict

Apollo cannot override discovery on company assignment.

## Contact memory

The contact phase should have its own memory, separate from company memory.

Save at least:

- discovered people;
- the company they were tied to;
- the matched role key;
- the evidence URLs;
- the verdict;
- timestamps;
- enriched contact channels from Apollo;
- Apollo mismatches and failures.

This matters because the same companies and roles will be searched repeatedly.

Example reuse cases:

- a later run again asks for estimators at the same company;
- a later run asks for managers at the same set of companies;
- a person already discovered last week can be reused if still fresh and still in scope.

## Freshness model for contacts

Contact data goes stale faster than company data.

So the memory policy should be stricter.

Recommended defaults:

- discovery memory freshness: 30 days
- Apollo enrichment freshness: 14 days

Past that point, the tool should prefer a fresh check.

## Proposed CLI shape

```bash
leads contacts discover --spec contact_search_spec.json
leads contacts enrich CONTACT_DISCOVERY_RUN_ID
leads contacts show-run CONTACT_DISCOVERY_RUN_ID
leads contacts inspect CONTACT_DISCOVERY_RUN_ID --person "Jane Smith"
leads contacts export CONTACT_DISCOVERY_RUN_ID
leads contacts show-enrichment CONTACT_ENRICHMENT_RUN_ID
leads contacts inspect-enrichment CONTACT_ENRICHMENT_RUN_ID --person "Jane Smith"
leads contacts export-enrichment CONTACT_ENRICHMENT_RUN_ID
```

## Contact discovery command behavior

`leads contacts discover --spec contact_search_spec.json`

Should do this:

1. validate the contact spec
2. load companies from the referenced enrichment run
3. optionally restrict to selected domains
4. check contact memory first
5. calculate missing role/company gaps
6. generate search queries per company and role
7. run Exa
8. normalize person candidates
9. evaluate current company and role fit
10. save accepted / review / rejected contacts
11. export artifacts

## Contact enrichment command behavior

`leads contacts enrich CONTACT_DISCOVERY_RUN_ID`

Should do this:

1. load accepted contacts from contact discovery
2. reuse fresh Apollo enrichment memory when available
3. search Apollo for the discovered person
4. match Apollo candidates against discovered identity
5. save email, phone, Apollo URL or record reference, and mismatch notes
6. export ready / review / blocked results

## How Apollo should be used

Apollo should behave like a constrained enrichment provider.

Its job is to answer:

- can we find a work email for this person?
- can we find a phone number?
- what does Apollo currently say about the person?

Its job is not:

- deciding whether the person currently works there;
- deciding whether the role is correct;
- deciding whether the company is in scope.

## Apollo matching rules

The Apollo lookup should start from the discovered person record, not from an open-ended search.

Use:

- full name
- company name
- company domain
- role/title
- LinkedIn URL if discovered

Possible Apollo outcomes:

### Ready

- Apollo record matches person identity strongly
- company matches target company or target domain
- email or phone is present

### Review

- identity likely matches but company field is stale or ambiguous
- email is present but company mismatch exists
- multiple Apollo records are plausible

### Blocked

- no credible Apollo match
- Apollo only returns clearly different people
- Apollo data conflicts too strongly to trust

## Very important Apollo trust rules

Do not silently trust Apollo if:

- the company differs from the discovered company;
- the role differs materially from the discovered role;
- the record looks old or stale;
- the email domain does not fit the discovered company.

Examples:

- if Apollo gives `jane@acmebuilders.com` for Jane Smith, that is strong
- if Apollo gives `jane@oldco.com`, that is not usable for Acme Builders
- if Apollo gives a personal Gmail, that should usually go to review
- if Apollo gives a phone with stale company data, keep it only with explicit review status

## Recommended final output fields

The client-facing export should stay small and contain:

- `company_name`
- `company_domain`
- `contact_name`
- `title`
- `linkedin_url`
- `email`
- `phone`
- `status`
- `notes`

Discovery writes the same shape with blank email and phone fields. Later enrichment can fill those
columns without changing the client-facing contract.

Detailed role keys, current-company checks, Apollo metadata, individual field statuses, review
flags, source URLs, evidence, and reasoning belong in `run.json` or review artifacts rather than
the final CSV.

## Current implementation status

Both phases are implemented as separate commands. Contact enrichment consumes only accepted people
from a completed contact discovery run, batches Apollo requests by 10, polls asynchronous request
IDs, applies deterministic identity and company/channel checks, reuses fresh 14-day Apollo memory,
and exports ready/review/blocked artifacts under the source contact run.

The default command requests email and phone and therefore requires `APOLLO_WEBHOOK_URL`. The
email-only fallback is `leads contacts enrich CONTACT_DISCOVERY_RUN_ID --no-phone`.

## Artifacts

Contact discovery should export:

- `accepted.csv`
- `review.csv`
- `rejected.csv`
- `summary.md`
- `run.json`

Contact enrichment should export:

- `ready.csv`
- `review.csv`
- `blocked.csv`
- `summary.md`
- `run.json`

## Skills needed

The natural-language flow should be supported by at least two new skills:

### 1. Contact search spec writer

Responsibility:

- turn the user's request into `contact_search_spec.json`
- choose the source enrichment run
- resolve whether the user means all companies or only some named companies
- convert raw title language into structured role targets

### 2. Contact discovery operator

Responsibility:

- run `leads contacts discover`
- explain the run scope
- summarize accepted/review/rejected results
- help inspect uncertain people

### 3. Contact enrichment operator

Responsibility:

- run `leads contacts enrich`
- explain that Apollo is enrichment-only
- summarize ready/review/blocked contact outputs
- flag stale-company mismatches explicitly

Validate:

Persist:

- evidence URLs

Goal:

- reuse recent people findings before hitting the web again.

## Phase 4: Build Exa-based contact search

Implement:

- query planner per company and role

- produce a strong candidate pool from live sources.

Implement a structured evaluator that decides:

- does this person currently work at the company?
- does the title match the requested role?

Goal:

- make current-role judgment explicit and inspectable.

## Phase 6: Export contact discovery artifacts

Produce:

Implement:

- person lookup from discovered identity

## Phase 9: Add CLI commands and progress UX

- loading company scope
- checking contact memory

Use distinct visual sections for:

- live web discovery
- Apollo enrichment

The user should always understand which stage the tool is in and why.

## Final recommendation

The cleanest design is:

- companies discover which companies fit
- companies enrich complete the company profile
- contacts discover which people currently fit
- contacts enrich use Apollo to fetch channels for those people

This preserves one strong rule across the whole tool:

> Use current evidence first for selection, then use structured providers second for enrichment.

That rule is exactly what protects the system from Apollo's stale employee data.
